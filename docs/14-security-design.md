## 1. 보안 설계 원칙

### 1.1 3대 핵심 원칙

| 원칙 | 설명 | 구현 방침 |
|------|------|----------|
| **로컬 처리 우선** | 모든 탐지 및 분석은 사용자 기기 내에서 수행 | 서버 전송 없음 (Phase 1), 네트워크 통신은 OUI DB 업데이트만 |
| **최소 권한** | 앱 기능에 반드시 필요한 권한만 요청 | 각 권한의 사용 이유를 사용자에게 명시적으로 설명 |
| **데이터 미수집** | 사용자 개인정보를 수집하거나 외부로 전송하지 않음 | 분석 데이터 로컬 저장, 익명 통계조차 Phase 1에서 수집하지 않음 |

### 1.2 보안 설계 계층

```
┌─────────────────────────────────────────────────┐
│              Layer 4: 코드 보안                  │
│  ProGuard/R8 난독화, 디버그 빌드 분리, 무결성   │
├─────────────────────────────────────────────────┤
│              Layer 3: 데이터 보안                │
│  SQLCipher 암호화, 민감도 분류, 삭제 정책        │
├─────────────────────────────────────────────────┤
│              Layer 2: 네트워크 보안              │
│  로컬 스캔 범위 제한, HTTPS 통신, 인증서 고정    │
├─────────────────────────────────────────────────┤
│              Layer 1: 권한 보안                  │
│  최소 권한 요청, 런타임 권한, 사용 후 해제        │
└─────────────────────────────────────────────────┘
```

### 1.3 Zero Trust 로컬 원칙

Phase 1에서 SearCam은 외부 서버와 통신하지 않는다. 따라서 네트워크 기반 공격 벡터가 원천 차단된다.

- 사용자 계정 없음 (로그인 불필요)
- 원격 API 호출 없음 (탐지 로직 100% 온디바이스)
- 원격 데이터 저장 없음 (모든 리포트 로컬 Room DB)
- 외부 SDK 최소화 (AdMob 광고만, 스캔 중 미동작)

## 2. 위협 모델링 (STRIDE 분석)

### 2.1 STRIDE 프레임워크 적용

SearCam의 주요 자산과 데이터 흐름에 대해 STRIDE 6개 카테고리로 위협을 분석한다.

#### 2.1.1 자산 식별

| 자산 | 민감도 | 설명 |
|------|--------|------|
| 네트워크 기기 MAC 주소 | 중간 | 같은 네트워크 기기의 하드웨어 식별자 |
| Wi-Fi SSID/BSSID | 중간 | 네트워크 식별 정보 |
| 카메라 프레임 데이터 | 높음 | 실시간 영상 (메모리에서만 처리) |
| 자기장 측정값 | 낮음 | 센서 수치 (위치 추론 불가) |
| 스캔 리포트 | 중간 | 탐지 결과 + 기기 목록 + 위험도 |
| OUI 데이터베이스 | 낮음 | 공개 데이터 (IEEE 기반) |

#### 2.1.2 STRIDE 위협 매트릭스

| 위협 유형 | 위협 시나리오 | 대상 자산 | 위험도 | 대응 |
|-----------|-------------|----------|--------|------|
| **S**poofing (위장) | 악성 앱이 SearCam UI를 위장하여 사용자 속임 | 앱 자체 | 낮음 | 앱 서명 검증, Play Store 유일 배포 |
| **S**poofing | 공격자가 카메라 MAC 주소를 일반 기기로 위장 | MAC 주소 | 중간 | 포트 스캔 + 서비스 탐색 교차 검증 |
| **T**ampering (변조) | 로컬 DB의 스캔 리포트를 변조 | 스캔 리포트 | 낮음 | SQLCipher 암호화, 무결성 해시 |
| **T**ampering | OUI DB를 변조하여 카메라를 안전 기기로 표시 | OUI DB | 중간 | 앱 내장 DB + 업데이트 시 서명 검증 |
| **R**epudiation (부인) | 사용자가 스캔 결과를 부인하여 법적 분쟁 | 스캔 리포트 | 낮음 | 리포트에 타임스탬프 + 기기 해시 포함 |
| **I**nformation Disclosure (정보 노출) | 스캔 중 수집한 MAC 주소가 외부로 유출 | MAC 주소 | 중간 | 메모리에서만 처리, 리포트에 해시된 MAC만 저장 |
| **I**nformation Disclosure | 카메라 프레임 데이터 유출 | 카메라 프레임 | 높음 | 프레임 저장 안 함, 메모리 즉시 해제 |
| **D**enial of Service (서비스 거부) | 대량 네트워크 기기로 스캔 지연/크래시 | 앱 가용성 | 중간 | 기기 수 상한 (254대), 타임아웃 적용 |
| **E**levation of Privilege (권한 상승) | 앱 권한을 이용한 다른 앱/시스템 공격 | 시스템 | 낮음 | 최소 권한, 센서 사용 후 즉시 해제 |

### 2.2 네트워크 스캔 시 보안 위험

#### 2.2.1 포트 스캔 법적/기술적 이슈

포트 스캔은 네트워크 보안에서 민감한 영역이다. SearCam의 포트 스캔은 다음 범위로 제한한다.

| 항목 | 범위 제한 |
|------|----------|
| 대상 | 사용자가 연결된 동일 로컬 네트워크(같은 서브넷)만 |
| 포트 | 카메라 관련 포트만 (554, 80, 8080, 8888, 3702, 1935) |
| 방식 | TCP Connect (SYN 스캔 아님, 비공격적) |
| 속도 | 1기기당 최대 6포트, 동시 5개 연결 (네트워크 부하 최소화) |
| 타임아웃 | 포트당 2초 (응답 없으면 즉시 종료) |
| 기록 | 스캔 대상 IP는 리포트에 저장하지 않음 (결과만 저장) |

#### 2.2.2 네트워크 스캔 윤리 원칙

```
1. 사용자의 명시적 동의 하에만 스캔 (스캔 버튼 누를 때)
2. 동일 네트워크(같은 Wi-Fi)에 연결된 기기만 스캔
3. 외부 네트워크 접근 시도 절대 금지
4. 발견된 기기 정보를 외부로 전송하지 않음
5. 스캔 결과에서 카메라 의심 기기 외 일반 기기 상세 정보 표시하지 않음
```

### 2.3 카메라 데이터 유출 위험

| 위협 | 대응 |
|------|------|
| 카메라 프레임이 앱 외부로 전송 | 프레임은 메모리에서만 분석, 저장/전송 코드 없음 |
| 스크린 캡처로 프레임 유출 | OS 수준 방어는 불가, 사용자 책임 영역 |
| 메모리 덤프에서 프레임 추출 | 분석 완료 후 즉시 메모리 해제 (bitmap.recycle()) |
| 렌즈 감지 중 주변 영상 기록 | 영상 녹화 기능 없음, 실시간 분석만 수행 |

### 2.4 리포트 데이터 보호

리포트에 포함되는 데이터와 민감도 분류는 다음과 같다.

| 리포트 항목 | 민감도 | 보호 수준 |
|------------|--------|----------|
| 스캔 일시 | 낮음 | 평문 저장 |
| 스캔 장소명 (사용자 입력) | 중간 | SQLCipher 암호화 |
| 발견 기기 수 | 낮음 | 평문 저장 |
| 의심 기기 정보 (제조사, 포트) | 중간 | SQLCipher 암호화 |
| MAC 주소 | 중간 | SHA-256 해시 후 저장 (원본 미저장) |
| IP 주소 | 중간 | 리포트에 미포함 (스캔 시에만 사용) |
| 종합 위험도 점수 | 낮음 | 평문 저장 |
| 레이어별 상세 점수 | 낮음 | 평문 저장 |

## 3. 권한 관리 상세

### 3.1 필요 권한 목록

| 권한 | 레벨 | 사용 이유 | 사용 시점 | 최소화 전략 |
|------|------|----------|----------|------------|
| `ACCESS_FINE_LOCATION` | Dangerous | Wi-Fi 스캔 시 Android 요구 (API 26+) | Quick/Full Scan | 스캔 완료 후 위치 수집 중단 |
| `ACCESS_WIFI_STATE` | Normal | Wi-Fi 연결 상태 확인 | 스캔 시작 전 | 상태 확인만, 스캔 데이터 미수집 |
| `CHANGE_WIFI_STATE` | Normal | Wi-Fi 스캔 트리거 (startScan) | Quick/Full Scan | 스캔 완료 후 비활성화 |
| `CAMERA` | Dangerous | 렌즈 Retroreflection + IR 감지 | 렌즈 찾기, Full Scan | 분석 완료 후 카메라 즉시 해제 |
| `INTERNET` | Normal | OUI DB 업데이트, AdMob 광고 | 백그라운드 업데이트 | 업데이트 시에만 사용 |
| `VIBRATE` | Normal | 의심 기기 발견 시 진동 알림 | 스캔 중 | 사용자 설정으로 비활성화 가능 |
| `FLASHLIGHT` | Normal | 플래시 Retroreflection 렌즈 감지 | 렌즈 찾기 모드 | 렌즈 감지 모드에서만 ON |

### 3.2 권한 미사용 목록 (명시적 제외)

다음 권한은 의도적으로 요청하지 않는다.

| 미사용 권한 | 이유 |
|------------|------|
| `READ_CONTACTS` | 개인정보와 무관, 탐지 기능에 불필요 |
| `READ_PHONE_STATE` | 전화 기능 불필요 |
| `RECORD_AUDIO` | 음성 녹음 불필요 (카메라 영상만 분석) |
| `ACCESS_BACKGROUND_LOCATION` | 백그라운드 위치 불필요 |
| `BLUETOOTH_SCAN` | Phase 1에서 BLE 스캔 미구현 |
| `READ_EXTERNAL_STORAGE` | 외부 저장소 접근 불필요 |
| `WRITE_EXTERNAL_STORAGE` | PDF 내보내기는 SAF(Storage Access Framework) 사용 |

### 3.3 런타임 권한 요청 플로우

```
앱 최초 실행
  │
  ▼
[온보딩 화면 1/3]
  "SearCam이 몰래카메라를 탐지하기 위해 아래 권한이 필요합니다"
  │
  ▼
[온보딩 화면 2/3]  ──→ "건너뛰기" (권한 없이 체크리스트만 사용)
  "Wi-Fi 스캔을 위해 위치 권한이 필요합니다"
  "→ 위치 정보를 수집하거나 저장하지 않습니다"
  │
  ├─ [허용] → 위치 권한 시스템 다이얼로그
  │           ├─ 허용 → Wi-Fi 스캔 기능 활성화
  │           └─ 거부 → "렌즈 감지 + 자기장만 사용 가능합니다"
  │
  ▼
[스캔 버튼 클릭 시 — 카메라 권한]
  "렌즈 탐지를 위해 카메라 권한이 필요합니다"
  "→ 영상을 녹화하거나 저장하지 않습니다"
  │
  ├─ [허용] → 렌즈 감지(Retroreflection + IR) 활성화
  └─ [거부] → "Wi-Fi + 자기장만 사용 가능합니다"
```

### 3.4 권한 거부 시 기능 매핑

| 거부된 권한 | 사용 가능 기능 | 사용 불가 기능 |
|------------|--------------|---------------|
| 위치 | 렌즈 찾기, IR, EMF, 체크리스트 | Quick Scan, Full Scan의 Wi-Fi 스캔 |
| 카메라 | Quick Scan, EMF, 체크리스트 | 렌즈 찾기, IR 감지 |
| 위치 + 카메라 모두 거부 | EMF, 체크리스트 | Wi-Fi 스캔, 렌즈 감지 |

## 4. 데이터 보호

### 4.1 로컬 DB 암호화 (SQLCipher)

```
┌───────────────────────────────────────────────┐
│                Room DB + SQLCipher             │
│                                               │
│  암호화 알고리즘: AES-256-CBC                  │
│  키 파생: PBKDF2-HMAC-SHA512 (256,000 iter)   │
│  페이지 크기: 4096 bytes                       │
│  키 저장: Android Keystore (TEE/SE 지원)       │
│                                               │
│  ┌─────────────┐   ┌─────────────┐           │
│  │ reports     │   │ settings    │           │
│  │ (암호화)    │   │ (암호화)    │           │
│  └─────────────┘   └─────────────┘           │
│                                               │
│  키 생성 시점: 앱 최초 실행                    │
│  키 접근: 앱 프로세스만 (Keystore 바인딩)      │
│  루팅 기기: 경고 표시 후 사용 가능             │
└───────────────────────────────────────────────┘
```

### 4.2 리포트 데이터 민감도 분류

| 분류 | 데이터 | 보호 조치 |
|------|--------|----------|
| **높음** | MAC 주소 (원본) | 메모리에서만 처리, 저장 시 SHA-256 해시 |
| **높음** | 카메라 프레임 | 저장 안 함, 메모리 즉시 해제 |
| **중간** | 스캔 장소명 | SQLCipher 암호화 저장 |
| **중간** | 의심 기기 제조사/포트 | SQLCipher 암호화 저장 |
| **낮음** | 위험도 점수/등급 | 앱 내부 DB에만 저장 |
| **낮음** | 자기장 측정값 | 실시간 표시 후 요약값만 리포트에 저장 |

### 4.3 데이터 삭제 정책

| 데이터 | 보관 기간 | 삭제 방식 |
|--------|----------|----------|
| 카메라 프레임 | 0초 (메모리 즉시 해제) | bitmap.recycle() + GC |
| ARP/MAC 데이터 | 스캔 완료까지 (메모리) | 스캔 종료 시 null 할당 |
| 스캔 리포트 (무료) | 최근 10건 유지 | FIFO, 11번째 저장 시 가장 오래된 리포트 삭제 |
| 스캔 리포트 (프리미엄) | 사용자 수동 삭제 | 개별 삭제 + 전체 삭제 기능 제공 |
| 내보낸 PDF | 사용자 파일시스템 | 앱 관할 아님 (SAF) |
| 앱 삭제 시 | 전체 | Android 시스템이 앱 데이터 완전 삭제 |

### 4.4 데이터 생명주기

```
[스캔 시작]
    │
    ▼
[센서 데이터 수집] ──→ 메모리에서만 처리
    │                    (MAC, 프레임, 자기장 원시값)
    │
    ▼
[분석 엔진] ──→ 교차 검증, 위험도 산출
    │
    ▼
[결과 요약 생성] ──→ 민감 데이터 제거/해시
    │
    ▼
[로컬 DB 저장] ──→ SQLCipher 암호화
    │
    ▼
[원시 데이터 삭제] ──→ 메모리 해제 (null + GC)
```

## 5. 네트워크 보안

### 5.1 포트 스캔의 법적/윤리적 고려

| 고려사항 | SearCam 대응 |
|---------|-------------|
| 같은 네트워크 내 포트 스캔의 합법성 | 사용자가 접속한 네트워크 내에서만 수행, 외부 네트워크 스캔 불가 |
| ISP/공유기에 의한 스캔 차단 | TCP Connect 방식으로 비공격적, 소량 포트만 대상 |
| 탐지 대상의 프라이버시 | 카메라 의심 기기만 상세 분석, 일반 기기는 제조사/유형만 표시 |
| 네트워크 부하 | 동시 연결 5개, 포트당 2초 타임아웃, 총 스캔 30초 이내 |
| 기업 네트워크 보안 정책 | 스캔이 차단되면 "네트워크 정책으로 스캔이 제한됩니다" 안내 |

### 5.2 OUI DB 업데이트 시 HTTPS 통신

```
┌─────────────┐     HTTPS/TLS 1.3     ┌──────────────┐
│  SearCam    │ ───────────────────→  │  CDN 서버     │
│  (클라이언트)│ ←───────────────────  │  (OUI DB)     │
│             │   서명된 JSON 응답     │               │
└─────────────┘                       └──────────────┘

보안 조치:
1. TLS 1.3 필수 (TLS 1.2 이하 거부)
2. Certificate Pinning (인증서 고정)
   - 서버 인증서 공개키 해시를 앱에 내장
   - MITM 공격 방지
3. 응답 검증
   - JSON 구조 검증 (스키마 유효성)
   - 서명 검증 (Ed25519)
   - 버전 체크 (다운그레이드 방지)
4. 업데이트 주기
   - 앱 시작 시 1회 확인 (Wi-Fi 연결 시에만)
   - 마지막 업데이트 후 7일 경과 시에만 요청
   - 실패 시 기존 로컬 DB 사용 (서비스 중단 없음)
```

### 5.3 네트워크 통신 정책

### 6.1 ProGuard/R8 난독화

```
# proguard-rules.pro (릴리스 빌드)

# 기본 난독화
-obfuscationdictionary proguard-dict.txt
-classobfuscationdictionary proguard-dict.txt
-packageobfuscationdictionary proguard-dict.txt

# 탐지 알고리즘 핵심 클래스 보호
-keep class com.searcam.data.analysis.** { *; }
-keepclassmembers class com.searcam.data.analysis.CrossValidator {
    private <fields>;
}

# OUI 데이터베이스 구조 보호
-keep class com.searcam.data.analysis.OuiDatabase { *; }

# Room DB 엔티티 유지 (리플렉션 사용)
-keep class com.searcam.data.local.** { *; }

# 로그 제거
-assumenosideeffects class android.util.Log {
    public static int d(...);
    public static int v(...);
    public static int i(...);
}
```

### 6.2 디버그 빌드 vs 릴리스 빌드

| 항목 | 디버그 빌드 | 릴리스 빌드 |
|------|-----------|-----------|
| 난독화 | 비활성화 | R8 full mode |
| 로그 출력 | Timber.d() 활성화 | Timber.d() 제거 (ProGuard) |
| SQLCipher | 비밀번호: "debug" | Android Keystore 키 |
| 네트워크 인증서 | 시스템 CA 신뢰 | 인증서 고정 (pinning) |
| 디버거 연결 | 허용 | 거부 (isDebuggable=false) |
| ADB 백업 | 허용 | 거부 (allowBackup=false) |
| 스택 트레이스 | 전문 표시 | 난독화된 트레이스 (매핑 파일 보관) |

### 6.3 API 키 관리 (Phase 2+)

Phase 1에서는 외부 API를 사용하지 않으므로 API 키가 없다. Phase 2 이후:

| 키 종류 | 저장 위치 | 접근 방식 |
|--------|----------|----------|
| AdMob App ID | AndroidManifest.xml (공개 가능) | 빌드 시 주입 |
| OUI 업데이트 서버 키 | local.properties (Git 제외) | BuildConfig로 주입 |
| Firebase 키 (Phase 3) | google-services.json | CI/CD 시크릿 |
| 서명 키스토어 | CI/CD 환경 변수 | GitHub Actions Secrets |

```kotlin
// BuildConfig를 통한 키 주입 (절대 소스코드에 하드코딩하지 않음)
// build.gradle.kts
buildTypes {
    release {
        buildConfigField("String", "OUI_UPDATE_URL",
            "\"${project.findProperty("OUI_UPDATE_URL") ?: ""}\"")
    }
}
```
