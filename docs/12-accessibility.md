## 1. 접근성 목표 및 원칙

### 1.1 목표

SearCam은 **모든 사용자**가 안전을 확인할 수 있어야 합니다. 시각, 청각, 운동, 인지 장애를 가진 사용자도 핵심 기능(스캔, 결과 확인, 신고)을 독립적으로 수행할 수 있도록 설계합니다.

### 1.2 준수 기준

| 기준 | 레벨 | 적용 |
|------|------|------|
| WCAG 2.1 AA | 필수 | Phase 1 MVP |
| WCAG 2.1 AAA | 권장 | 일부 항목 (색상 대비 등) |
| Android Accessibility Guidelines | 필수 | Phase 1 |
| Material Design 3 Accessibility | 참고 | 디자인 시스템 |

### 1.3 핵심 원칙 (POUR)

| 원칙 | 설명 | SearCam 적용 |
|------|------|-------------|
| **Perceivable** (인식 가능) | 모든 정보를 사용자가 인식할 수 있어야 함 | 텍스트 대안, 색상+형태 조합, 음성 안내 |
| **Operable** (조작 가능) | 모든 UI를 조작할 수 있어야 함 | 큰 터치 타겟, 키보드 접근, 시간 제한 없음 |
| **Understandable** (이해 가능) | 정보와 조작이 이해 가능해야 함 | 쉬운 한국어, 일관된 네비게이션, 오류 안내 |
| **Robust** (견고) | 다양한 보조 기술과 호환 가능해야 함 | Compose 접근성 API, TalkBack 테스트 |

### 2.2 색상 외 시각 표현

색상만으로 정보를 전달하지 않습니다. 항상 색상 + 형태/텍스트를 조합합니다.

| 정보 | 색상만 | 색상 + 대안 (SearCam) |
|------|--------|---------------------|
| 위험 등급 | 초록/노랑/빨강 | 색상 + 텍스트 라벨 ("안전", "주의", "위험") + 아이콘 |
| 스캔 진행 상태 | 색상 변화 | 색상 + 체크마크/스피너/원 아이콘 + 텍스트 |
| 기기 위험도 | 빨간 배지 | 색상 배지 + 텍스트 라벨 + 아이콘 |
| 게이지 | 그라데이션 | 게이지 + 중앙 숫자 + 하단 텍스트 등급 |

## 3. 청각 접근성

### 3.1 청각 정보의 시각적/촉각적 대안

SearCam은 소리로 전달하는 모든 정보에 대해 시각적/촉각적 대안을 제공합니다.

| 청각 정보 | 시각적 대안 | 촉각적 대안 |
|---------|----------|----------|
| 스캔 완료 효과음 | 화면 전환 + 체크 아이콘 애니메이션 | 짧은 진동 (200ms) |
| 의심 포인트 경고음 | 빨간 원 펄스 + 화면 깜빡임 | 연속 진동 (100ms x 3) |
| EMF 비프음 (자기장 강도) | 막대 그래프 크기 변화 + 색상 변화 | 진동 강도 비례 |
| 위험 알림 | 배너 노출 + 빨간 배경 | 강한 진동 (500ms) |
| 스캔 진행 중 틱 소리 | 원형 진행률 + 카운트다운 텍스트 | 매초 미세 진동 |

### 3.2 진동 피드백 체계

| 이벤트 | 진동 패턴 | 강도 | 설정 |
|--------|---------|------|------|
| 스캔 시작 | 단일 (100ms) | 중 | 기본 ON |
| 기기 발견 | 단일 (50ms) | 약 | 기본 ON |
| 의심 포인트 감지 | 트리플 (100ms-50ms-100ms) | 강 | 기본 ON |
| 스캔 완료 | 더블 (150ms-50ms-150ms) | 중 | 기본 ON |
| 위험 결과 | 연속 (200ms x 5) | 강 | 기본 ON |
| 안전 결과 | 단일 (200ms) | 약 | 기본 ON |
| 에러 | 트리플 장음 (300ms-100ms-300ms) | 강 | 기본 ON |

### 3.3 설정

- 진동 ON/OFF (기본: ON)
- 진동 강도 (약/중/강)
- 소리 ON/OFF (기본: ON)
- 소리 볼륨 (시스템 미디어 볼륨 따름)

## 5. 인지 접근성

### 5.1 단순한 언어

#### 용어 정책

| 기술 용어 | 사용자 표시 | 비고 |
|---------|----------|------|
| ARP 스캔 | Wi-Fi 스캔 | 기본 UI |
| mDNS/SSDP | 기기 탐색 | 기본 UI |
| MAC OUI | 제조사 확인 | 기본 UI |
| Retroreflection | 렌즈 찾기 | 기본 UI |
| EMF (Electromagnetic Field) | 자기장 | 기본 UI, 약어 미사용 |
| RTSP | 영상 스트리밍 | 상세 화면에서만 |
| μT (마이크로 테슬라) | μT | 상세 화면에서만, 처음 등장 시 "자기장 단위" 주석 |

#### 문장 작성 규칙

| 규칙 | 예시 |
|------|------|
| 짧은 문장 (20자 이내 권장) | "이상이 감지되지 않았습니다" (13자) |
| 능동형 ("~합니다") | "스캔을 시작합니다" (o) / "스캔이 시작됩니다" (x) |
| 이중 부정 금지 | "카메라가 없을 수 없습니다" (x) → "카메라가 있을 수 있습니다" (o) |
| 전문 약어 미사용 | "EMF 스캔" (x) → "자기장 스캔" (o) |
| 행동 안내 명확 | "문제가 발생했습니다" (x) → "다시 시도해주세요" (o) |

### 5.2 단계별 안내

복잡한 기능은 단계별로 안내합니다.

#### 렌즈 찾기 모드 단계별 가이드

```
단계 1/4: 플래시를 켜세요
  "자동으로 플래시가 켜집니다.
   수동으로 끄지 마세요."
   [자동 진행]

단계 2/4: 천천히 비추세요
  "벽면과 천장을 천천히 비춰주세요.
   속도: 초당 30cm 이하"
   [안내 애니메이션]

단계 3/4: 의심 포인트 확인
  "빨간 원이 나타나면
   해당 위치를 가까이에서 확인하세요."
   [자동 감지]

단계 4/4: 결과 확인
  "스캔이 완료되었습니다.
   결과를 확인하세요."
   [스캔 완료 버튼]
```

### 5.3 오류 방지 및 복구

| 상황 | 오류 방지 | 복구 방법 |
|------|---------|---------|
| 리포트 삭제 | "삭제하시겠습니까?" 확인 다이얼로그 | "취소" 버튼 (3초 이내 되돌리기) |
| 권한 거부 | 왜 필요한지 설명 후 재요청 | 설정 앱으로 이동 링크 |
| 스캔 중 이탈 | "스캔을 중단하시겠습니까?" 확인 | "계속하기" / "중단" 선택 |
| 네트워크 오류 | 오류 원인 설명 + 해결 방법 | "Wi-Fi를 확인하고 다시 시도해주세요" |

    )

### 7.2 커스텀 컴포넌트 접근성

#### RiskGauge (위험도 게이지)

```kotlin
@Composable
fun RiskGauge(
    score: Int,
    riskLevel: RiskLevel,
    modifier: Modifier = Modifier
) {
    Box(
        modifier = modifier.semantics {
            // TalkBack 읽기: "종합 위험도 72점, 위험 등급"
            contentDescription = "종합 위험도 ${score}점, ${riskLevel.label} 등급"
            // ProgressBar 역할 부여
            role = Role.Image
            // 상태 정보
            stateDescription = "${riskLevel.label}"
        }
    ) {
        // 시각적 게이지 그리기
        Canvas(modifier = Modifier.fillMaxSize()) {
            // ... 호(arc) 그리기
        }
        // 중앙 점수 텍스트
        Text(
            text = "$score",
            modifier = Modifier.semantics {
                // 개별 읽기 방지 (부모에서 통합 읽기)
                clearAndSetSemantics { }
            }
        )
    }
}
```

#### ScanProgress (스캔 진행)

```kotlin
@Composable
fun ScanProgress(
    progress: Float,
    remainingSeconds: Int,
    currentStep: String,
    modifier: Modifier = Modifier
) {
    Column(
        modifier = modifier.semantics(mergeDescendants = true) {
            contentDescription = "스캔 진행 중, " +
                "${(progress * 100).toInt()}% 완료, " +
                "${remainingSeconds}초 남음. " +
                "현재 단계: $currentStep"
            // LiveRegion: 상태 변경 시 자동 알림
            liveRegion = LiveRegionMode.Polite
        }
    ) {
        // UI 요소들
    }
}
```

### 7.3 LiveRegion (상태 변경 자동 알림)

스캔 중 실시간으로 변하는 정보는 LiveRegion을 설정하여 TalkBack이 자동으로 읽어줍니다.

```kotlin
// 스캔 결과 변경 시 자동 알림
Text(
    text = "의심 포인트: ${pointCount}개",
    modifier = Modifier.semantics {
        liveRegion = LiveRegionMode.Polite // 현재 읽기 완료 후 알림
    }
)

// 위험 감지 시 즉시 알림
Text(
    text = "위험 감지!",
    modifier = Modifier.semantics {
        liveRegion = LiveRegionMode.Assertive // 즉시 알림 (중단)
    }
)
```

### 7.4 포커스 관리

```kotlin
// 스캔 완료 시 결과에 포커스 이동
val focusRequester = remember { FocusRequester() }

LaunchedEffect(scanState) {
    if (scanState == ScanState.COMPLETED) {
        // 결과 텍스트로 포커스 이동
        focusRequester.requestFocus()
    }
}

Text(
    text = "종합 위험도: $score",
    modifier = Modifier.focusRequester(focusRequester)
)
```

## 8. 테스트 계획

### 8.1 자동 테스트

| 도구 | 용도 | 실행 시점 |
|------|------|---------|
| Accessibility Scanner (Google) | 터치 타겟, 대비, 라벨 자동 검사 | 매 빌드 |
| Espresso Accessibility Checks | UI 테스트에 접근성 검사 통합 | CI/CD |
| Lint (Android Studio) | contentDescription 누락 경고 | 코딩 중 |

### 8.2 수동 테스트

| 테스트 | 방법 | 빈도 |
|--------|------|------|
| TalkBack 전체 탐색 | 모든 화면을 TalkBack으로 탐색 + 조작 | 주 1회 |
| Switch Access 테스트 | 외부 스위치로 모든 기능 접근 | 월 1회 |
| 폰트 크기 최대 테스트 | 시스템 폰트 200% + 표시 확대 | 매 릴리즈 |
| 색상 반전 테스트 | 시스템 색상 반전 모드에서 사용성 확인 | 매 릴리즈 |
| 그레이스케일 테스트 | 흑백 모드에서 정보 구별 가능 여부 | 매 릴리즈 |

### 8.3 사용자 테스트

| 대상 | 인원 | 방법 |
|------|------|------|
| 시각 장애 사용자 (TalkBack 이용) | 2명 | 1:1 사용성 테스트 (60분) |
| 저시력 사용자 | 2명 | 1:1 사용성 테스트 (60분) |
| 운동 장애 사용자 | 1명 | Switch Access 사용성 테스트 |
