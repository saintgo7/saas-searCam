## 1. 국제화 전략 개요

### 1.1 목표

SearCam은 한국 시장을 시작으로, 일본과 글로벌 영어권 시장으로 확장합니다. 국제화(i18n)는 앱 아키텍처 단계부터 설계하여, 새로운 언어 추가 시 코드 변경 없이 리소스 파일만 추가하면 되는 구조를 목표로 합니다.

### 1.2 지원 언어 로드맵

| Phase | 언어 | 시기 | 비고 |
|-------|------|------|------|
| **Phase 1** (MVP) | 한국어 (ko) | 출시 시 | 기본 언어, 전체 지원 |
| **Phase 2** | 영어 (en) | MVP + 3개월 | 글로벌 기본, 에어비앤비 여행객 타겟 |
| **Phase 3** | 일본어 (ja) | MVP + 6개월 | 일본 도촬 문제, 높은 수요 예상 |
| **Phase 4** (검토) | 중국어 간체 (zh-CN) | MVP + 9개월 | 중국 관광객 대상, 법적 검토 필요 |
| **Phase 5** (검토) | 태국어 (th), 베트남어 (vi) | MVP + 12개월 | 동남아 숙박 시설 수요 |

### 1.3 지원 범위

| 요소 | Phase 1 | Phase 2+ |
|------|---------|---------|
| UI 텍스트 | 한국어만 | 다국어 |
| 숫자 포맷 | 한국 (1,234) | 로케일별 |
| 날짜 포맷 | 한국 (2026.04.03) | 로케일별 |
| 통화 | 원 (₩) | 로케일별 |
| 체크리스트 콘텐츠 | 한국어 | 번역 |
| 법적 고지 | 한국 법률 기준 | 각국 법률 검토 |
| 스토어 메타데이터 | 한국어 | 다국어 ASO |

### 2.3 복수형 처리 (plurals.xml)

한국어는 단수/복수 구분이 없지만, 영어와 일본어를 위해 plurals 리소스를 사용합니다.

```xml
<?xml version="1.0" encoding="utf-8"?>
<resources>
    <!-- 한국어: 복수형 구분 없음 -->
    <plurals name="devices_found">
        <item quantity="other">발견된 기기: %d대</item>
    </plurals>

    <plurals name="suspect_points">
        <item quantity="other">의심 포인트: %d개</item>
    </plurals>

    <plurals name="reports_count">
        <item quantity="other">%d건</item>
    </plurals>

    <plurals name="checklist_items">
        <item quantity="other">%d항목</item>
    </plurals>
</resources>
```

**영어 복수형 (values-en/plurals.xml)**:

```xml
<?xml version="1.0" encoding="utf-8"?>
<resources>
    <plurals name="devices_found">
        <item quantity="one">%d device found</item>

    </plurals>
```

#### Kotlin 구현

```kotlin
import java.time.LocalDateTime
import java.time.format.DateTimeFormatter
import java.time.format.FormatStyle
import java.util.Locale

fun formatScanDate(dateTime: LocalDateTime, locale: Locale): String {
    val formatter = DateTimeFormatter.ofLocalizedDateTime(
        FormatStyle.MEDIUM,
        FormatStyle.SHORT
    ).withLocale(locale)
    return dateTime.format(formatter)
}

// 사용
formatScanDate(scanDate, Locale.KOREA)   // "2026. 4. 3. 오후 2:32"
formatScanDate(scanDate, Locale.US)      // "Apr 3, 2026, 2:32 PM"
formatScanDate(scanDate, Locale.JAPAN)   // "2026/04/03 14:32"
```

## 3. 번역 워크플로우

### 3.1 번역 프로세스

```
1. 개발자: strings.xml (한국어) 작성
   │
2. 키 등록: 번역 플랫폼에 업로드
   │
3. 번역가: 각 언어별 번역 수행
   │
4. 리뷰어: 번역 품질 검수 (네이티브 스피커)
   │
5. 개발자: 번역 파일 다운로드 → values-{locale}/ 배치
   │
6. QA: pseudo-localization 테스트 + 실제 언어 테스트
   │
7. 배포
```

### 3.2 번역 도구 선택

| 도구 | 장점 | 단점 | 채택 |
|------|------|------|------|
| Crowdin | Android XML 네이티브 지원, GitHub 연동 | 유료 (오픈소스 무료) | Phase 2+ |
| Lokalise | 개발자 친화적, CI/CD 연동 | 비용 높음 | 검토 |
| POEditor | 저렴, 기본 기능 충분 | 고급 기능 부족 | 검토 |
| 수동 관리 | 비용 0 | 관리 어려움, 누락 위험 | Phase 1 |

**Phase 1**: 한국어만이므로 수동 관리.
**Phase 2+**: Crowdin 도입하여 번역 워크플로우 자동화.

### 3.3 번역 가이드라인 (번역가용)

#### 톤 & 보이스

| 항목 | 규칙 |
|------|------|
| 존댓말 수준 | 한국어: 해요체 ("~합니다", "~해주세요") |
|  | 영어: 직접적, 친절 ("Tap to scan", "No threats found") |
|  | 일본어: 정중체 ("~します", "~してください") |
| 기술 용어 | 번역하지 않는 용어: Wi-Fi, IR, EMF, RTSP, MAC, IP |
| 브랜드명 | "SearCam"은 번역하지 않음 (모든 언어 동일) |
| 문장 길이 | 원문 길이의 130% 이내 (UI 깨짐 방지) |

#### 번역하지 않는 문자열


## 4. 문화적 고려사항

### 4.1 시장별 몰래카메라 인식

#### 한국 (Phase 1)

| 항목 | 내용 |
|------|------|
| 사회적 인식 | 매우 높음. 주요 사회 이슈. 뉴스 보도 빈번 |
| 법적 환경 | 성폭력범죄의 처벌 등에 관한 특례법. 최대 7년 징역 |
| 주요 발생 장소 | 공중화장실, 숙박업소, 탈의실, 에스컬레이터 |
| 신고 채널 | 112 (경찰), 여성긴급전화 1366 |
| 용어 | 몰래카메라, 몰카, 불법촬영 |
| 앱 내 대응 | 112 신고 버튼, 한국 특화 체크리스트 (원룸, 고시원 등) |

#### 일본 (Phase 3)

| 항목 | 내용 |
|------|------|
| 사회적 인식 | 높음. "도촬" (盗撮) 사회 문제 |
| 법적 환경 | 각 도도부현 조례로 규제. 도쿄: 최대 1년 징역 또는 100만엔 벌금 |
| 주요 발생 장소 | 공중목욕탕 (온천/센토), 호텔, 전철 |
| 신고 채널 | 110 (경찰) |
| 용어 | 盗撮カメラ, 隠しカメラ |
| 앱 내 대응 | 110 신고 버튼, 온천/료칸 체크리스트 추가 |

#### 영어권 - 글로벌 (Phase 2)

| 항목 | 내용 |
|------|------|
| 사회적 인식 | 증가 중. Airbnb 숨겨진 카메라 이슈로 주목 |
| 법적 환경 | 미국: 주(State)별 상이. 영국: Voyeurism Act 2019 |

## 5. RTL (Right-to-Left) 지원 계획

### 5.1 현재 지원 범위

Phase 1~3 대상 언어(한국어, 영어, 일본어)는 모두 LTR(Left-to-Right)입니다. RTL은 아랍어, 히브리어 등 지원 시 필요합니다.

### 5.2 RTL 준비 설계 (선제적)

코드 변경 없이 RTL을 지원할 수 있도록, 처음부터 RTL 호환 레이아웃을 설계합니다.

#### Compose에서의 RTL 대응

```kotlin
// RTL 호환: start/end 사용 (left/right 사용 금지)
Row(
    modifier = Modifier.padding(start = 16.dp, end = 16.dp) // left/right 대신
) {
    // ...
}

// RTL 호환: Arrangement 사용
Row(
    horizontalArrangement = Arrangement.Start // Left 대신
) {
    // ...
}
```

#### RTL 금지 패턴

| 금지 | 대안 |
|------|------|
| `paddingLeft` / `paddingRight` | `paddingStart` / `paddingEnd` |
| `marginLeft` / `marginRight` | `marginStart` / `marginEnd` |
| `Alignment.Left` | `Alignment.Start` |
| `drawableLeft` | `drawableStart` |
| 하드코딩된 방향 아이콘 | 미러링 가능한 벡터 아이콘 |

#### AndroidManifest.xml

```xml
<application
    android:supportsRtl="true"
    ... >
```

### 5.3 RTL 미러링 대상/비대상

| 요소 | 미러링 | 이유 |
|------|--------|------|
| 네비게이션 화살표 (← →) | 미러링 | 방향성 |
| 체크박스 위치 | 미러링 | 읽기 흐름 |
| 진행 방향 (프로그레스바) | 미러링 | 읽기 흐름 |
| 시계/타이머 | 미러링 안 함 | 보편적 |
| 미디어 재생 버튼 (▶) | 미러링 안 함 | 보편적 |
| 브랜드 로고 | 미러링 안 함 | 브랜드 아이덴티티 |
| 카메라 프리뷰 | 미러링 안 함 | 실제 방향 |

## 7. 테스트 전략

### 7.1 Pseudo-Localization (가짜 번역)

실제 번역 전에 UI가 다국어를 수용할 수 있는지 테스트합니다.

#### 테스트 방법

```
원문:  "스캔 중..."
Pseudo: "[Ṡĉàñ ïñ þŕöĝŕëšš... оди|два]"
```

- 원문의 각 문자를 유사 문자로 대체 (악센트, 특수 문자)
- 텍스트 길이를 30~50% 확장 (독일어/프랑스어 대비)
- 대괄호로 감싸서 잘림 감지
- 키릴 문자 혼합으로 인코딩 문제 감지

#### Android Studio 설정

```gradle
// build.gradle.kts
android {
    buildTypes {
        debug {
            // Pseudo-localization 활성화
            isPseudoLocalesEnabled = true
        }
    }
}
```

설정 → 언어에서 "XA (Pseudo)" 또는 "XB (RTL Pseudo)" 선택하여 테스트.

### 7.2 번역 품질 테스트

| 테스트 항목 | 방법 | 기준 |
|-----------|------|------|
| 텍스트 잘림 | 모든 화면 스크린샷 비교 | 잘리는 텍스트 0건 |
| 텍스트 겹침 | Layout Inspector | 겹치는 요소 0건 |
| 번역 누락 | strings.xml 키 비교 스크립트 | 누락 키 0건 |
| 맞춤법/문법 | 네이티브 리뷰어 검수 | 오류 0건 |
| 용어 일관성 | 용어집(glossary) 대조 | 불일치 0건 |
| 문화적 적절성 | 현지인 사용성 테스트 | 부적절 표현 0건 |

### 7.3 누락 키 검출 스크립트

```bash
#!/bin/bash
# check_missing_translations.sh
# 기본 언어(ko) 대비 번역 누락 키를 검출

BASE_DIR="app/src/main/res"
DEFAULT_LANG="values"   # 한국어 (기본)
TARGET_LANGS=("values-en" "values-ja")

for lang in "${TARGET_LANGS[@]}"; do
    echo "=== Checking $lang ==="

    for xml_file in "$BASE_DIR/$DEFAULT_LANG"/strings*.xml; do
        filename=$(basename "$xml_file")
        target_file="$BASE_DIR/$lang/$filename"

        if [ ! -f "$target_file" ]; then
            echo "  MISSING FILE: $filename"
            continue
        fi

        # 기본 파일의 키 추출 (translatable="false" 제외)
        default_keys=$(grep -oP 'name="[^"]*"' "$xml_file" | \
                       grep -v 'translatable="false"' | \
                       sort)
        target_keys=$(grep -oP 'name="[^"]*"' "$target_file" | sort)

        # 차이점 확인
        missing=$(comm -23 <(echo "$default_keys") <(echo "$target_keys"))
        if [ -n "$missing" ]; then
            echo "  MISSING in $filename:"
            echo "$missing" | sed 's/^/    /'
        fi
    done
done
```
