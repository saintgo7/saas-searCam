## 1. 평가 개요

### 1.1 평가 목적

본 개인정보 영향평가(Privacy Impact Assessment, PIA)는 SearCam 앱이 처리하는 데이터의 프라이버시 위험을 사전에 식별하고, 위험 완화 조치를 수립하여 사용자의 개인정보를 보호하기 위해 작성한다.

SearCam은 "개인정보를 수집하지 않는다"는 설계 원칙을 채택하고 있으나, 앱 기능상 카메라, 네트워크, 센서 데이터를 처리하므로 잠재적 프라이버시 위험이 존재한다. 본 평가는 이러한 잠재적 위험을 체계적으로 분석한다.

### 1.2 평가 범위

| 항목 | 범위 |
|------|------|
| 대상 시스템 | SearCam Android 앱 (Phase 1 MVP) |
| 평가 대상 데이터 | 네트워크 기기 정보, 카메라 프레임, 센서 데이터, 스캔 리포트 |
| 제외 대상 | Phase 2 iOS 앱, Phase 3 서버/커뮤니티 기능 (별도 평가 필요) |
| 평가 기준 | 개인정보 보호법, 정보통신망법, GDPR (참조), NIST Privacy Framework |

### 1.3 평가 방법론

본 평가는 다음 절차에 따라 수행한다.

```
1. 데이터 흐름 분석
   └─ 앱이 처리하는 모든 데이터의 수집/사용/저장/삭제 흐름 매핑

2. 프라이버시 원칙 점검
   └─ 데이터 최소화, 목적 제한, 투명성, 보안 원칙 준수 확인

3. 위험 식별 및 평가
   └─ 잠재적 프라이버시 위험 식별, 확률/영향 평가

4. 완화 조치 수립
   └─ 각 위험에 대한 기술적/관리적 완화 조치 설계

5. 잔여 위험 수용
   └─ 완화 후 남은 위험의 수용 가능 여부 판단

6. 정기 재평가 계획
   └─ 앱 업데이트/기능 추가 시 재평가 일정
```

### 1.4 평가 주체

| 역할 | 담당 |
|------|------|
| 평가 수행자 | 개발팀 + 보안 담당 |
| 검토자 | PM |
| 승인자 | 프로젝트 오너 |

## 2. 시스템 개요

### 2.1 시스템 목적

SearCam은 일반인이 숙소, 화장실, 탈의실 등에서 스마트폰만으로 숨겨진 카메라를 탐지할 수 있는 1차 스크리닝 도구이다.

### 2.2 주요 기능

| 기능 | 데이터 처리 | 프라이버시 관련성 |
|------|-----------|----------------|
| Wi-Fi 네트워크 스캔 | MAC 주소, IP, 포트 정보 | 높음 (네트워크 기기 식별) |
| 렌즈 Retroreflection 감지 | 카메라 프레임 (실시간) | 높음 (주변 영상 처리) |
| IR 야간 카메라 감지 | 전면 카메라 프레임 | 높음 (주변 영상 처리) |
| 자기장(EMF) 감지 | 3축 자력계 데이터 | 낮음 (개인 식별 불가) |
| 교차 검증 | 3개 레이어 분석 결과 | 중간 (종합 판단) |
| 리포트 저장 | 스캔 결과 요약 | 중간 (장소/기기 정보 포함) |
| 체크리스트 | 사용자 입력 (체크 상태) | 낮음 |

### 2.3 이해관계자

| 이해관계자 | 역할 | 프라이버시 관심사 |
|-----------|------|-----------------|
| 앱 사용자 | 몰래카메라 탐지 | 자신의 데이터가 외부로 유출되지 않는지 |
| 같은 네트워크의 다른 사용자 | 간접 영향 | 자신의 기기가 스캔되어 정보가 수집되는지 |
| 숙소/시설 운영자 | 간접 영향 | 네트워크 기기 정보가 노출되는지 |
| 앱 개발자 | 서비스 제공 | 법적 의무 준수 |
| 광고 네트워크 (Google AdMob) | 광고 제공 | 광고 ID, 기기 정보 수집 |

## 3. 데이터 흐름 분석

### 3.1 수집 데이터

#### 3.1.1 Wi-Fi 기기 MAC 주소 (임시 처리)

```
┌────────────────────────────────────────────────────────┐
│ 데이터: MAC 주소 (예: 28:57:BE:xx:xx:xx)              │
│                                                        │
│ 수집 방법: /proc/net/arp 읽기 (Android)                │
│ 처리 목적: 카메라 제조사 OUI 매칭                       │
│ 저장 형태: 메모리에서만 처리, SHA-256 해시 후 리포트 저장│
│ 보관 기간: 스캔 완료까지 (메모리), 해시값은 리포트에 저장│
│ 외부 전송: 없음                                        │
│                                                        │
│ [수집] ─→ [OUI 매칭] ─→ [해시 변환] ─→ [리포트 저장]  │
│   ↓                                        ↓           │
│ 원본 삭제 (메모리 해제)          SHA-256 해시만 보관     │
└────────────────────────────────────────────────────────┘

개인정보 해당 여부:
- MAC 주소 원본: 기기 식별 가능 → 개인정보 해당 가능
- SHA-256 해시: 원본 복원 불가 → 비식별 정보
- SearCam 조치: 원본은 메모리에서만 처리하고 즉시 삭제
```

#### 3.1.2 카메라 프레임 (메모리만 처리)

```
┌────────────────────────────────────────────────────────┐
│ 데이터: 카메라 영상 프레임 (30fps, 720p)               │
│                                                        │
│ 수집 방법: CameraX API (후면/전면 카메라)              │
│ 처리 목적: 렌즈 Retroreflection 분석, IR LED 감지      │
│ 저장 형태: 저장하지 않음 (메모리 버퍼에서 실시간 분석)  │
│ 보관 기간: 프레임당 약 33ms (다음 프레임 도착 시 교체)  │
│ 외부 전송: 없음                                        │
│                                                        │
│ [카메라 프레임] ─→ [고휘도 포인트 추출] ─→ [패턴 분석] │
│       ↓                                       ↓        │
│  즉시 교체/해제                        결과(점수)만 보존│
│                                        프레임 자체 폐기│
└────────────────────────────────────────────────────────┘

개인정보 해당 여부:
- 카메라 프레임은 주변 환경(방, 벽, 가구)을 촬영
- 사람이 촬영될 수 있으나, 프레임을 저장하지 않으므로
  "수집"에 해당하지 않음
- 실시간 처리 후 즉시 폐기
```

#### 3.1.3 자기장 수치 (로컬 처리)

```
┌────────────────────────────────────────────────────────┐
│ 데이터: 3축 자기장 벡터 (x, y, z μT)                  │
│                                                        │
│ 수집 방법: SensorManager (TYPE_MAGNETIC_FIELD, 20Hz)   │
│ 처리 목적: 전자기기 존재 감지                           │
│ 저장 형태: 실시간 표시, 요약값(최대 변화량)만 리포트 저장│
│ 보관 기간: 리포트 보관 기간과 동일                      │
│ 외부 전송: 없음                                        │
│                                                        │
│ [센서 데이터] ─→ [노이즈 필터] ─→ [임계값 판정] ─→    │
│       ↓                                    ↓           │
│  실시간 그래프 표시                 요약값 리포트 저장   │
│  (화면 전환 시 폐기)              (점수, 최대 변화량)   │
└────────────────────────────────────────────────────────┘

개인정보 해당 여부:
- 자기장 수치는 물리량 데이터
- 개인을 식별하거나 추적하는 데 사용할 수 없음
- 개인정보에 해당하지 않음
```

```
┌──────────────────────────────────────────┐
│                                          │
│  SearCam 앱                              │
│  ┌────────────────────────┐              │
│  │  탐지 엔진             │              │

│  ┌────────────────────────┐   HTTPS      │  ┌──────────┐
│  │  OUI 업데이트 모듈     │ ──────────→  │  │ CDN 서버  │
│  │  (수신 전용)           │ ←──────────  │  │ (JSON)   │
│  └────────────────────────┘              │  └──────────┘
│                                          │
│  ┌────────────────────────┐   SDK 통신   │  ┌──────────┐
│  │  AdMob SDK             │ ──────────→  │  │ Google   │
│  │  (Google 관리)         │ ←──────────  │  │ AdMob    │
│  └────────────────────────┘              │  └──────────┘
│                                          │
└──────────────────────────────────────────┘

전송 데이터 상세:
1. OUI 업데이트: 버전 번호만 전송 → OUI JSON 수신 (사용자 데이터 없음)
2. AdMob: Google SDK가 자체적으로 처리 (광고 ID, 기기 정보)
   → SearCam 코드에서 사용자 데이터를 AdMob에 전달하지 않음
   → Google 개인정보 처리방침 적용
```

### 3.4 전체 데이터 흐름도

```
[사용자가 스캔 시작]
       │
       ├─── Layer 1: Wi-Fi ──→ ARP 테이블 ──→ MAC 주소 (메모리)
       │                            │
       │                       OUI 매칭 + 포트 스캔
       │                            │
       │                       SHA-256 해시 ──→ 리포트
       │                            │
       │                       원본 MAC 삭제 (메모리 해제)
       │
       ├─── Layer 2: 렌즈 ──→ 카메라 프레임 (메모리 버퍼)
       │                            │
       │                       고휘도 포인트 분석
       │                            │
       │                       분석 결과(점수) ──→ 리포트
       │                            │
       │                       프레임 폐기 (bitmap.recycle)
       │
       ├─── Layer 3: EMF ──→ 자기장 데이터 (실시간)
       │                            │
       │                       노이즈 필터 + 판정
       │                            │
       │                       요약값 ──→ 리포트
       │
       └─── 교차 검증 ──→ 종합 위험도 산출
                               │
                          리포트 저장 (SQLCipher)
                               │
                          화면에 결과 표시
```

## 4. 프라이버시 원칙 점검

### 4.1 데이터 최소화 원칙

| 점검 항목 | 상태 | 근거 |
|----------|------|------|
| 기능에 필요한 최소한의 데이터만 처리하는가? | 준수 | MAC 주소는 OUI 매칭에만 사용, 카메라 프레임은 분석에만 사용 |
| 불필요한 데이터를 수집하지 않는가? | 준수 | GPS 좌표, 전화번호, 연락처 등 미수집 |
| 데이터 보관 기간이 최소화되어 있는가? | 준수 | 센서 데이터 즉시 폐기, 리포트 10건 제한 |

### 4.2 목적 제한 원칙

| 점검 항목 | 상태 | 근거 |
|----------|------|------|
| 데이터가 명시된 목적에만 사용되는가? | 준수 | 탐지 목적에만 사용, 마케팅/분석 미사용 |
| 목적 외 이용이 불가능한 구조인가? | 준수 | 서버 미사용, 데이터 외부 전송 없음 |
| 사용자에게 목적이 투명하게 공개되는가? | 준수 | 권한 요청 시 사용 이유 명시 |

### 4.3 투명성 원칙

| 점검 항목 | 상태 | 근거 |
|----------|------|------|
| 처리하는 데이터 종류를 사용자가 알 수 있는가? | 준수 | 개인정보 처리방침, 앱 내 안내 |
| 데이터 처리 방식을 사용자가 이해할 수 있는가? | 준수 | 온보딩, 권한 요청 화면에서 설명 |
| 사용자가 데이터 삭제를 요청할 수 있는가? | 준수 | 리포트 개별/전체 삭제, 앱 삭제 시 전량 삭제 |

### 4.4 보안 원칙

| 점검 항목 | 상태 | 근거 |
|----------|------|------|
| 저장 데이터가 암호화되어 있는가? | 준수 | SQLCipher AES-256 |
| 전송 데이터가 암호화되어 있는가? | 준수 | OUI 업데이트 HTTPS/TLS 1.3 |
| 접근 제어가 적용되어 있는가? | 준수 | Android Keystore, 앱 샌드박스 |

## 5. 위험 식별 및 평가

### 5.1 위험 평가 기준

| 등급 | 확률 | 영향도 |
|------|------|--------|
| 높음 | 발생 가능성 50% 이상 | 대규모 프라이버시 침해 |
| 중간 | 발생 가능성 10~50% | 소규모 영향, 복구 가능 |
| 낮음 | 발생 가능성 10% 미만 | 영향 미미, 즉시 해결 가능 |

### 5.2 위험 목록

#### P-01: MAC 주소 원본 메모리 노출

| 항목 | 내용 |
|------|------|
| 위험 | 메모리 덤프 공격으로 분석 중인 MAC 주소 원본 추출 |
| 확률 | 낮음 (루팅 기기 + 전문 도구 필요) |
| 영향도 | 중간 (같은 네트워크 기기의 MAC 주소 노출) |
| **위험 등급** | **낮음** |
| 완화 조치 | 분석 완료 즉시 null 할당 + GC, 루팅 기기 경고 |

#### P-02: 카메라 프레임에 사람 포함

| 항목 | 내용 |
|------|------|
| 위험 | 렌즈 감지 중 촬영된 프레임에 타인이 포함되어 프라이버시 침해 |
| 확률 | 중간 (숙소에서는 1인 사용이 일반적이나, 공공 장소 가능) |
| 영향도 | 중간 (프레임 저장 안 하므로 실제 유출 가능성 극히 낮음) |
| **위험 등급** | **중간** |
| 완화 조치 | 프레임 저장 기능 없음, 분석 후 즉시 폐기, 앱 UI에 "영상을 저장하지 않습니다" 고지 |

#### P-03: 스캔 리포트에서 장소 정보 추론

## 7. 잔여 위험 수용 근거

### 7.1 수용 가능한 잔여 위험

| 잔여 위험 | 수용 근거 |

현재 평가에서 수용 불가능한 잔여 위험은 식별되지 않았다.

- 모든 완화 조치가 설계대로 구현되었을 것
- 릴리스 전 보안 검토에서 프라이버시 관련 결함이 없을 것
- 개인정보 처리방침과 데이터 안전 섹션이 정확히 공개되었을 것

## 9. 결론

### 9.1 평가 결과 요약

SearCam Phase 1 (Android MVP)은 다음과 같은 프라이버시 설계 원칙에 따라 개인정보 위험이 매우 낮은 수준이다.

| 원칙 | 준수 상태 |
|------|----------|
| 로컬 처리 우선 | 완전 준수 — 모든 탐지 로직 온디바이스 실행 |
| 데이터 미수집 | 완전 준수 — 개인정보 서버 전송 없음 |
| 최소 권한 | 완전 준수 — 기능에 필요한 권한만 요청 |
| 데이터 최소화 | 완전 준수 — 센서 데이터 즉시 폐기, 리포트 최소 정보만 저장 |
| 투명성 | 완전 준수 — 데이터 처리 방식 사용자에게 명시 |
| 암호화 | 완전 준수 — SQLCipher + HTTPS |

### 9.2 총평

SearCam Phase 1은 "데이터를 수집하지 않는" 설계 원칙 덕분에 프라이버시 위험이 본질적으로 낮다. 11개 식별된 위험 중 완화 후 잔여 위험이 "중간"인 항목은 P-04 (AdMob SDK)뿐이며, 이는 업계 표준 관행으로 수용 가능하다.

Phase 2 (iOS), Phase 3 (서버/커뮤니티)에서는 데이터 처리 범위가 확대되므로, 해당 시점에서 반드시 PIA를 전면 재수행해야 한다.
